What is Docker?

"A way to pack, ship and run any application as a lightweight container"
- From the Docker website
A way for developers to package any application to run on (almost) any infrastructure

What is a Docker image?

  • Consistent, standard way of packaging applications
  • Similar to Unix chroot
  • Includes all dependencies, configuration files, etc.
  • Published to and installed from a registry
  • Snapshot of a container

What is a Docker container?

  • An instance of a running image
  • Isolated set of resources
    • Processes
    • Networking
    • Filesystem
  • Can run on multiple runtimes: libcontainer, LXC, KVM, BSD Jails, Solaris Zones, etc
  • Lighter (and faster to start) than a VM

Why Docker?

Dependency and environment management

  • An image contains all dependencies
    • JVM
    • Libraries (including native)
    • Scripts
    • Configuration files
  • ...and is created from a text description
    • Infrastructure as code
    • Environment controlled in VCS

Standardized infrastructure

  • Linux distributions and packages
  • No more "it works on my machine"
  • No more "it works on QA but not in production"

Immutable infrastructure

  • All changes are stored in source control
  • No hot-fixes that are not included in all hosts
  • No ad-hoc package updates

Ephemeral infrastructure

  • Do not upgrade, redeploy!
  • Machines should be able to come and go

    What is a Docker image?

    • A self-contained filesystem built in layers
      • Basic OS commands and system files (/bin, /etc, /lib, /var, /usr, ...)
      • JVM
      • Application artifacts
      • Application config files
    • Built in layers using copy-on-write
      • Base image
      • Application-specific files

    How to build a Docker image

    1. Create a Dockerfile
      • Set the base image
      • Add layers
        • Additional packages
        • Application artifacts
        • Set which command to run
        • Which user to run as
        • ...
    2. Build the image
    3. Deploy to a registry

    Docker registries

    • Similar to a GitHub repository
    • Central place to store (versioned) images
    • Public (usually free) and private (usually for-pay)
    • Default is http://hub.docker.com
    • Can install your own (https://github.com/docker/distribution and https://docs.docker.com/registry/)

    Caveats

    Images should be ephemeral and immutable

    • Use centralized logging
    • Use network data sources
    • Configuration files should be included at build time
    • Dynamic configuration should be read from env variables

    Workaround (not recommended): Map a local directory to a filesystem in the container

    Networking

    Docker images listen on a virtual net interface
    • Can't talk to the outside world
    • Can't talk to other containers on same host

    Solutions
      --link to connect containers
      --publish to expose specific ports
      --net=host to expose all ports
      Docker 1.9 adds docker network for added flexibility

    https://docs.docker.com/v1.8/userguide/dockerlinks/ https://blog.docker.com/2015/11/docker-multi-host-networking-ga/

    Dockerizing a Scala application

    Selecting a base image

    • java:openjdk-8
    • Official java image
    • Based on Ubuntu LTS and OpenJDK
    • JDK and JRE-based for Java 6, 7, 8 and 9
    • https://hub.docker.com/_/java/

    Selecting a base image

    There are other Java images using other Linux distributions (Alpine, CentOS). Some of them also include the Oracle JDK/JRE. Browse https://hub.docker.com/search/?q=java

    Create a user, etc.

                  
                RUN /usr/bin/apt-get update && \
    /usr/bin/apt-get install -y locales git postgresql-client && \
    /usr/bin/apt-get clean && \
    /usr/sbin/locale-gen en_US en_US.UTF-8 && \
    /usr/sbin/useradd -r -s /bin/false -d /home/appuser appuser

    Add your code

    • Use COPY to add any .class or .jar files
    • It's easiest to use sbt-assembly or sbt-native-packager
      •               
                COPY target/scala-2.11/myApp.jar /home/appuser/lib

    Managing configuration files

    There are 3 options for adding your configuration files:
    • Different images per machine configuration
    • Single image, different configuration files
    • Single configuration file

    Configuration: Different images

    • One Dockerfile per configuration
    • One image per configuration
    Advantage: Configuration is completely static
    Disadvantages:
    • Multiple builds (therefore: time, can get out of sync)
    • Multiple images in repository (although can use tags)
    • Not easily implementable using sbt-docker

    Configuration: Single image, different config files

    • One single Dockerfile
    • Configurations stored in directories
    • Environment variable to select configuration at startup
    Advantage: Single build, single image
    Disadvantage: Can be more complex to manage, files can get out of sync, need env vars

    Configuration: Single config files

    • One single Dockerfile
    • Single config file, use env vars to adapt at runtime
    Advantage: Single build, single image, single file
    Disadvantage: Config file is more complex, not as easy to test, need env vars

    Configuring the image

                  
dockerfile in docker := {
  val artifact = (outputPath in assembly).value
  val baseDir = "/srv"
  val preInstall = Seq(
    "/usr/bin/apt-get update",
    "/usr/bin/apt-get install -y locales git postgresql-client",
    "/usr/bin/apt-get clean",
    "/usr/sbin/locale-gen en_US en_US.UTF-8",
    s"/usr/sbin/useradd -r -s /bin/false -d $baseDir appuser"
  ).mkString(" && ")

  new Dockerfile {
    from("java:openjdk-8-jre")
    runRaw(preInstall)
    workDir(baseDir)
    copy(new File("bin/run-docker.sh"), "bin/run.sh")
    copy(new File("local/etc"), "etc")
    runRaw("chmod 0544 bin/run.sh")
    copy(artifact, "lib/app.jar")
    user("appuser")
    entryPoint("bin/run.sh")
  }
}
